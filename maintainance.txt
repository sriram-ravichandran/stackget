 1. GitHub Upload & CI/CD

  Initialize and push:
  cd D:\Projects\Tools\StackGet
  git init
  git add .
  git commit -m "feat: initial release"
  git remote add origin https://github.com/sriram-ravichandran/stackget.git
  git branch -M main
  git push -u origin main

  To cut a release (triggers GoReleaser automatically):
  git tag v1.0.0
  git push origin v1.0.0

  GoReleaser will cross-compile all 6 targets (Windows/macOS/Linux × amd64/arm64), create .tar.gz archives, generate a changelog, and publish the GitHub release — all automatically. The ci.yml workflow also
  runs on every push/PR to catch build breaks early.

  2. npm Publishing

  One-time setup:
  1. npm login at https://www.npmjs.com
  2. Add NPM_TOKEN to your GitHub repo → Settings → Secrets → Actions
  3. Get it from [npmjs.com → Access Tokens → Generate New Token (Automation)]

  How it works: The npm/ directory contains the package. postinstall runs install.js which downloads the correct .tar.gz for the user's OS/arch from the GitHub release and extracts the binary into npm/bin/.
  The run.js wrapper then passes all CLI args through to that binary.

  After that, every git tag v1.x.x automatically:
  1. Builds all binaries via GoReleaser
  2. Publishes the npm package with the matching version

  Users install with: npm install -g stackget

  3. SOP — Adding a New Tool to tools.go

  Step 1: Find the right category in AllCategories. Categories are in logical order in the file.

  Step 2: Add a ToolDef struct to that category's Tools slice:

  // Minimal — works for any tool that follows --version convention
  {Name: "MyTool", Commands: []string{"mytool"}},

  // With explicit version args (tool uses -v, not --version)
  {Name: "MyTool", Commands: []string{"mytool"}, VersionArgs: []string{"-v"}},

  // When output contains unwanted version numbers (e.g. library versions)
  // Use VersionFilter to require a specific string in the output
  {Name: "Python 3", Commands: []string{"python3", "python"},
      VersionFilter: "Python 3"},

  // When semver regex picks up the wrong number
  // Use VersionRegex with exactly one capturing group
  {Name: "nmap", Commands: []string{"nmap"},
      VersionRegex: `Nmap version (\d+\.\d+)`},

  // Presence-only tool (no version flag exists)
  {Name: "kubectx", Commands: []string{"kubectx"}, NoVersion: true},

  // Tool that connects to a cluster — use --client to avoid hanging
  {Name: "kubectl", Commands: []string{"kubectl"},
      VersionArgs: []string{"version --client", "--version"}},

  // Slow-starting tool (Python-based, JVM-based)
  {Name: "HuggingFace CLI", Commands: []string{"huggingface-cli"},
      Timeout: 10 * time.Second},

  // GUI-only application (no CLI binary)
  {Name: "Postman", GUIApp: &GUIApp{
      WinExe:   "Postman.exe",
      MacApp:   "Postman.app",
      LinuxBin: "postman",
  }},

  Step 3: Verify it works:
  export PATH="/c/Program Files/Go/bin:$PATH"
  go build -o stackget.exe . && ./stackget.exe scan --only <category-name>

  Decision tree for VersionRegex: Run the tool manually (mytool --version) and check if the output contains version numbers that aren't the tool's own version. If yes → add VersionRegex. The group must capture
   exactly the version string you want.

  Decision tree for VersionFilter: If the same binary name can be multiple tools (e.g. python could be Python 2 or 3), set VersionFilter to a string that only appears in the correct variant's output.
